## 项目概述

### 项目背景

律社复刻是一个基于大语言模型的法律咨询 Agent 系统。我选择法律领域作为应用场景，主要考虑到法律文本结构严谨、知识体系清晰，非常适合用来实践 RAG 和 Agent 技术。

### 项目架构

项目采用前后端分离架构：
- **后端**：FastAPI + LangGraph + ChromaDB，提供 API 服务和核心 AI 能力
- **前端**：Streamlit 多页面应用，提供友好的用户交互界面
- **数据层**：ChromaDB 向量库 + Neo4j 图数据库 + Mem0 记忆系统

### 开发历程

整个开发过程历时约三周，我从最基础的 RAG 开始，逐步叠加 Agent、Skill、记忆、多智能体等能力。过程中遇到了不少困难，但每解决一个问题都收获很大。

## 任务2：Master Stacks 基础技术栈

### 任务要求

- 至少使用 1 种 Agent 框架
- 至少使用 1 种 LLM API 网关
- 至少使用 1 种向量数据库
- 选做：图数据库、文档解析器

### 技术选型

#### 1. Agent 框架：LangGraph

**为什么选 LangGraph**：
一开始我考虑过直接用 LangChain 的 Agent，但发现对于复杂的状态管理和多轮流程控制，LangChain 表达力不够。LangGraph 基于图的状态机模型非常适合构建有明确流程的 Agent 系统，特别是多智能体辩论这种需要循环和条件分支的场景。

**遇到的困难**：
- 刚开始学习 LangGraph 的状态模型时，对 StateGraph 的状态传递理解不深
- 节点之间的数据流转经常出错，特别是在循环节点中如何维护状态
- 调试比较困难，图的执行过程不像线性代码那样直观

**解决方案**：
- 先从简单的线性图开始练习，逐步增加条件边和循环
- 给每个节点的输入输出都加上日志，观察状态变化
- 参考官方的多智能体示例，理解 State 的设计模式

**核心实现**：
```python
# 多智能体辩论的状态图设计
graph = StateGraph(DebateState)
graph.add_node("researcher", _researcher_node)
graph.add_node("plaintiff", _plaintiff_node)
graph.add_node("defendant", _defendant_node)
graph.add_node("judge", _judge_node)

graph.add_edge(START, "researcher")
graph.add_edge("researcher", "plaintiff")
graph.add_edge("plaintiff", "defendant")
graph.add_edge("defendant", "judge")
graph.add_conditional_edges("judge", _should_continue, {
    "continue": "plaintiff",
    "end": END,
})
```

#### 2. LLM API 网关：LiteLLM

**为什么选 LiteLLM**：
项目需要支持多种大模型（千问、智谱等），如果每个模型都写一套调用代码会很冗余。LiteLLM 提供了统一的 OpenAI 兼容接口，切换模型只需要改模型名，非常方便。

**遇到的困难**：
- 不同模型的参数格式略有差异，比如 temperature 的范围
- 流式响应的处理在不同模型间表现不一致
- 错误处理需要统一封装

**解决方案**：
- 自己封装了 `LLMGateway` 类，统一了所有模型的调用方式
- 增加了 fallback 机制，主模型失败时自动切换到备用模型
- 统一的错误类型 `LLMGatewayError`，上层调用只需要捕获一种异常

#### 3. 向量数据库：ChromaDB

**为什么选 ChromaDB**：
作为入门项目，ChromaDB 足够轻量，不需要额外部署服务，直接本地持久化就行。API 也很简洁，上手快。

**遇到的困难**：
- Embedding 模型的选择，一开始想用 sentence-transformers，但 Python 3.14 兼容性有问题
- 中文分词效果影响检索质量
- 向量相似度和关键词匹配如何结合

**解决方案**：
- 先实现了一个基于 Hash 的轻量级 Embedding 作为 fallback，保证主流程能跑通
- 后续可以无缝替换为真正的 Embedding 模型
- 实现了 Hybrid Search：向量检索 + 关键词重排，兼顾语义和精确匹配

#### 4. 选做：图数据库 Neo4j

为了实践 Graph RAG，我集成了 Neo4j 作为图数据库。主要用来存储法律条文之间的引用关系，比如某条司法解释引用了哪个法条。

**实现文件**：`backend/core/graph_rag.py`

#### 5. 选做：文档解析器 Unstructured

合同审查需要解析 Word 和 PDF 文件，我集成了 Unstructured 作为文档解析器，支持本地模式和 API 模式两种。

**实现文件**：`backend/utils/document_parser.py`

## 任务3：Further RAG 进阶检索增强

### 任务要求

- 必须使用 Agentic RAG 方法
- 至少使用 3 种进阶 RAG 技术

### 实现的进阶 RAG 技术

#### 1. Query Transformation（查询改写）

**原理**：用户的问题往往口语化、不精准，直接检索效果不好。用 LLM 把用户问题改写成多个更适合检索的查询，然后分别检索再合并结果。

**遇到的困难**：
- 改写后的查询可能偏离原意
- 查询太多会增加延迟和 token 消耗

**优化方案**：
- 限制改写数量（最多 3 个）
- 增加 fallback 规则，LLM 不可用时用关键词拆分
- 结果去重，避免重复内容

#### 2. HyDE（假设性文档嵌入）

**原理**：先让 LLM 生成一个"可能的答案"，再用这个假设性答案去做向量检索。因为答案和知识库中的条文在语义空间更接近，检索效果往往比直接用问题检索好。

**实现思路**：
- 生成的假设答案不要求准确，只要求语义相近
- 控制生成长度，避免太长影响检索速度
- 和 Query Transformation 的结果合并，一起参与检索

#### 3. Context Enrichment（上下文增强）

**原理**：检索时用小块（chunk）保证精度，但生成回答时需要更多上下文。所以每个小块都保存它的父级上下文，生成时拼接回去。

**实现细节**：
- chunk_size=260，overlap=60
- parent_window=1，即每个 chunk 保存前后各一块的内容
- 检索返回小 chunk，生成时用 parent_text

#### 4. Hybrid Search（混合检索）

**原理**：纯向量检索擅长语义匹配，但对精确关键词不敏感；纯关键词检索相反。把两者结合，向量相似度 + 关键词得分加权求和，效果更好。

**打分公式**：
```
final_score = (1 / (1 + distance)) + keyword_score
```

#### 5. Graph RAG（图检索增强）

**原理**：法律条文之间有很多引用关系，用图数据库存储这些关系，检索时可以沿着关系扩展相关条文。

**遇到的困难**：
- Neo4j 的部署和配置花了不少时间
- 如何从法律文本中自动提取实体和关系是个难题
- 图查询的 Cypher 语法需要学习

**当前状态**：
- 完成了基础的 Graph RAG 框架
- 实体和关系抽取可以后续用 LLM 自动完成
- 作为选做功能，先把架子搭好

#### 6. Agentic RAG（Agent 驱动的检索）

这是任务要求的必选项。我的实现方式是：Agent 自主决定要不要检索、检索什么、检索几次，而不是固定的"检索-生成"两步骤。

**具体表现**：
- 自主 Agent 可以根据用户问题判断是否需要调用 RAG 工具
- 多智能体辩论中，每个角色自己决定检索什么关键词
- 检索结果不足时，可以换关键词再次检索

## 任务4：Autonomous Agents 自主智能体

### 任务要求

- 任务规划能力
- 文件读取/编辑
- 命令执行
- 安全机制（目录限制、手动/自动审批、沙箱执行）

### 核心实现

#### 1. 任务规划能力

Agent 收到复杂任务时，先把任务拆分成多个步骤，然后逐步执行。

**规划流程**：
1. 理解用户需求
2. 拆分子任务列表
3. 逐个执行子任务
4. 遇到问题动态调整计划
5. 汇总结果给出最终回答

**遇到的困难**：
- LLM 有时候规划的步骤太粗或者太细
- 执行过程中如何判断一个步骤完成了
- 计划变更时如何更新状态

**解决方案**：
- 用结构化输出（JSON）让 LLM 返回计划
- 每个步骤完成后做一次反思，确认是否达标
- 支持动态插入新步骤

#### 2. 文件读取/编辑

Agent 可以读取项目内的文件，也可以创建和修改文件。

**实现的工具**：
- `read_file`：读取文件内容
- `write_file`：写入新文件
- `edit_file`：修改文件内容
- `list_dir`：列出目录内容

#### 3. 命令执行

Agent 可以执行 shell 命令，比如运行测试、安装依赖等。

#### 4. 安全机制（重点）

这是我花心思最多的部分，Agent 直接操作文件和执行命令风险很大，必须有完善的安全机制。

**三级安全机制**：

1. **目录限制**
   - Agent 只能在指定的工作目录内操作
   - 禁止访问上级目录和系统目录
   - 路径规范化，防止 `../` 绕过

2. **命令白名单**
   - 只允许执行安全的命令（ls、cat、grep 等）
   - 危险命令（rm、sudo、curl 等）默认禁止
   - 可配置的白名单机制

3. **人工审批**
   - 高风险操作（文件写入、命令执行）需要用户确认
   - 支持自动审批模式（仅用于可信环境）
   - 每次操作都有日志记录

**安全模块文件**：`backend/core/safe_commands.py`

**遇到的困难**：
- 路径绕过的方式很多，比如符号链接、相对路径等
- Windows 和 Linux 的路径规则不一样
- 如何平衡安全性和易用性

**经验总结**：
安全机制宁可严格一点，也不要留漏洞。大模型有时候会"自作聪明"尝试绕过限制，必须在代码层面严格把关。

## 任务6：Agent Memory 记忆系统

### 任务要求

- 学习 mem0、MemOS、Letta 等框架
- 接入到现有项目

### 技术选型：Mem0

调研了几个记忆框架后，我选择了 Mem0，原因是：
1. 轻量级，集成简单
2. 支持多种存储后端
3. 有用户级别的记忆隔离
4. 中文支持还可以

### 实现方案

#### 三级记忆架构

参考人类记忆模型，设计了三级记忆：

1. **短期记忆**（对话上下文）
   - 当前会话的历史消息
   - 会话结束即消失

2. **工作记忆**（会话摘要）
   - 当前会话的关键信息提炼
   - 用于长对话中防止上下文丢失

3. **长期记忆**（Mem0 持久化）
   - 用户的偏好、历史咨询记录
   - 跨会话保留
   - 自动提取和更新

#### 记忆提取机制

- 每次对话后，自动判断是否有值得记忆的信息
- 用 LLM 提取关键事实，写入 Mem0
- 下次对话时，自动检索相关记忆作为上下文

**实现文件**：`backend/core/memory.py`

**遇到的困难**：
- 什么信息该记、什么不该记，很难把握
- 记忆太多会产生噪音，影响回答质量
- 记忆更新和过期的策略

**解决方案**：
- 只记忆用户明确表达的偏好和事实
- 每条记忆都有相关度评分，检索时取 Top N
- 后续可以增加记忆遗忘机制

## 任务8：Multi-Agent System 多智能体系统

### 任务要求

- 学习 Agno、CAMEL、CrewAI、elizaOS 等
- 实现自主多智能体系统

### 调研过程

我调研了几个多智能体框架：
- **CrewAI**：角色分工清晰，适合协作任务
- **CAMEL**：角色扮演式的通信，适合辩论场景
- **Agno**：功能全面，但比较重
- **elizaOS**：偏操作系统概念，比较新

最后决定**自己基于 LangGraph 实现**，而不是直接用框架。原因是：
1. 自己实现能更深入理解多智能体的原理
2. 法律辩论场景比较特殊，定制化需求高
3. LangGraph 已经提供了图编排能力，在此基础上搭多 Agent 正好

### 系统设计：专家会诊

我设计了一个"法律专家会诊"系统，三个角色：
- **原告代理人**：代表原告，提出诉讼主张
- **被告代理人**：代表被告，进行抗辩
- **法官**：中立裁判，评估双方论点，决定是否继续辩论

#### v1 版本：固定流程

最开始的版本是固定流程：
1. 检索法律资料
2. 原告陈述
3. 被告陈述
4. 法官评估
5. 未收敛则继续下一轮

#### v2 版本：修复轮数 Bug

开发中发现了一个 Bug：设置 5 轮但实际只有 2 轮。排查后发现是 fallback 模式里硬编码了 `current_round < 2`，完全忽略了 max_rounds 参数。

这个 Bug 藏得比较深，因为 LLM 模式下走不到 fallback 分支，只有降级时才触发。最后通过单测才发现。

#### v3 版本：子 Agent 自主检索（最新）

这是最大的一次升级，每个角色都变成了独立的子 Agent：

**核心改进**：
1. **自主 RAG 检索**：每个 Agent 自己去知识库找法律依据，不是统一检索后分发
2. **立场导向检索**：原告找支持原告的法条，被告找抗辩理由
3. **针对性反驳**：每轮必须针对对方上一轮的漏洞来组织反击
4. **自动类别识别**：根据案情自动判断属于哪类法律，只在相关类别中检索

**辩论逻辑**：
- 第一轮：原告提出主张，被告进行首轮抗辩
- 后续轮次：双方都针对对方上一轮的论点逐一反驳，找法律漏洞
- 法官每轮评估收敛度，达到阈值提前终止
- 最后法官给出判决意见

**遇到的困难**：
- 如何让 Agent 真的"针对性反驳"，而不是自说自话
- 收敛度的计算怎么设计才合理
- 多轮辩论容易重复，怎么保证每轮有新内容
- RAG 检索的结果如何有效融入辩论

**解决方案**：
- Prompt 里明确要求"逐一反驳对方论点"
- 收敛度用关键词重叠度 + 语义相似度综合计算
- 每轮要求提出新的法律依据，避免重复
- 把检索到的法条直接放进 Prompt，让 Agent 引用

## 开发心得与总结

### 遇到的主要困难

#### 1. 技术栈太多，学习曲线陡

一下子要学 LangGraph、LiteLLM、ChromaDB、Mem0、Playwright 这么多东西，刚开始有点应接不暇。

**应对方法**：先跑通最小可用版本，再逐个叠加功能。每个技术先掌握核心用法，不追求一开始就精通。

#### 2. 大模型的不确定性

LLM 的输出有时候不稳定，同样的 Prompt 可能得到不同的结果，调试起来很头疼。

**应对方法**：
- 增加 fallback 机制，LLM 不行就走规则
- 结构化输出（JSON）加解析容错
- 温度调低，增加稳定性

#### 3. RAG 效果调优

RAG 不是把文档塞向量库就完事了，检索质量受很多因素影响：chunk 大小、重叠数、Embedding 质量、重排策略等等。

**经验**：
- 中文场景下，纯向量检索效果不一定比关键词好
- 混合检索（向量+关键词）通常更稳
- Query Transformation 对口语化问题提升明显

#### 4. 多智能体的可控性

多 Agent 系统很容易"失控"，辩论可能跑偏，或者无限循环。

**经验**：
- 必须有明确的终止条件（最大轮数 + 收敛检测）
- 每个 Agent 的 Prompt 要严格限定角色和输出格式
- 法官角色很重要，负责把控方向
